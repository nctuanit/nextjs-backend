# Execution Pipeline: Guards, Interceptors, Pipes

The library operates a strict execution pipeline per-request:
1. **Guards**: Determines if the request is authorized.
2. **Interceptors**: Wraps the execution stream allowing pre/post execution logic.
3. **Pipes**: Validates and transforms input payload.
4. **Handler Method**: The actual controller logic.

## 1. Guards

A guard is a class annotated with the `@Injectable()` decorator, which implements the `CanActivate` interface. Let's create an `AuthGuard` that checks for a specific authorization header.

```typescript
import { CanActivate, Context, Injectable } from './src';

@Injectable()
export class AuthGuard implements CanActivate {
  async canActivate(context: Context): Promise<boolean> {
    const request = context.request;
    const token = request.headers.get('authorization');
    
    // In a real app, validate the JWT token here
    return token === 'Bearer my-secret-token';
  }
}
```

You can bind the guard to a specific route or an entire controller using `@UseGuards()`.

```typescript
import { Controller, Get, UseGuards } from './src';

@Controller('/secure')
@UseGuards(AuthGuard) // Protects all paths in this controller
export class SecureController {
  @Get('/data')
  getData() {
    return { sensitive: 'information' };
  }
}
```

## 2. Interceptors

Interceptors have a set of useful capabilities which are inspired by Aspect-Oriented Programming (AOP) techniques. They make it possible to:
- Bind extra logic before / after method execution
- Transform the result returned from a function
- Completely override a function depending on specific conditions (e.g., for caching)

Here is a `LoggingInterceptor` that intercepts the request flow.

```typescript
import { NextInterceptor, Context, Injectable } from './src';

@Injectable()
export class LoggingInterceptor implements NextInterceptor {
  async intercept(context: Context, next: () => Promise<unknown>) {
    console.log('Incoming request...');
    
    const now = Date.now();
    const result = await next(); // Proceed to Route Handler
    
    console.log(`Execution time: ${Date.now() - now}ms`);
    return result; 
  }
}
```

Apply it using `@UseInterceptors()`.

```typescript
@UseInterceptors(LoggingInterceptor)
@Get()
findAll() { return []; }
```

## 3. Pipes & Validation

Pipes have two typical use cases:
- **Transformation**: Transform input data to the desired form (e.g., from string to integer)
- **Validation**: Evaluate input data and if valid, simply pass it through unchanged; otherwise, throw an exception.

The library fully integrates with `class-validator` to mimic NestJS's strict and typed payload validation.

### DTO (Data Transfer Object)

Let's define a DTO schema using class-validator:

```typescript
import { IsString, IsEmail, MinLength } from 'class-validator';

export class CreateUserDto {
  @IsString()
  @MinLength(3)
  name: string;

  @IsEmail()
  email: string;
}
```

Now, bind it using the built-in `ValidationPipe`:

```typescript
import { Controller, Post, Body, UsePipes, ValidationPipe } from './src';

@Controller('/users')
export class UsersController {
  
  @Post()
  @UsePipes(new ValidationPipe())
  create(@Body() dto: CreateUserDto) {
    // \`dto\` is guaranteed to be a valid instance of CreateUserDto
    return { success: true, createdUser: dto };
  }
}
```

If the payload does not match the DTO constraints (e.g., `email` is structurally invalid), the pipe will automatically reject the request with an HTTP **400 Bad Request** Error indicating the validation issues.
